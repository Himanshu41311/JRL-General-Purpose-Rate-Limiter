Four algorithms. One interface.

Every policy picks one — the proxy dispatches to the matching Lua script atomically, so two requests arriving in the same instant can never both slip through a race condition.

01

Fixed window

A counter resets every N seconds. Cheapest option — can allow up to 2× the limit right at a window boundary.

02

Sliding window

A Redis sorted set tracks exact request timestamps. No boundary burst — the limit is always measured over the last N seconds, precisely.

03

Token bucket

Tokens refill continuously up to a capacity. Lets short bursts through immediately, then throttles to the steady refill rate.

04

Leaky bucket

A level fills per request and drains at a constant rate. Smooths bursts into a steady outflow instead of letting them through at once.

How a request actually gets decided

Three steps, all inside the request's own round trip — nothing here waits on Postgres.

01

Route + policy come from Redis

Every proxy instance reads the same route:{id} and route-policies:{id} keys — no per-instance staleness, no database round trip.

02

Policies evaluate in parallel

Each active policy runs its algorithm's Lua script against Redis concurrently. If any policy denies, the request is denied — most restrictive wins.

03

Allow and proxy, or deny with headers

Allowed requests stream straight through to your backend. Denied ones get 429 with Retry-After and X-RateLimit-* headers.
